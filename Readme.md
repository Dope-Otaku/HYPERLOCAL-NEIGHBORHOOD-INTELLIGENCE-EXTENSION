## Block 1 — Frontend & Extension Framework

The goal is a zero-friction developer experience that produces a Manifest V3 extension injecting cleanly into live real-estate portals without CSS collisions or DOM pollution.

### 1.1 Extension SDK — Plasmo Framework

Plasmo abstracts away the most painful parts of extension development: MV3 service-worker lifecycle, content-script injection rules, and HMR during development. Think of it as **Next.js for Chrome extensions**.

| Property | Detail |
|---|---|
| Language | TypeScript + React (JSX/TSX) |
| Manifest Version | V3 — auto-generated by Plasmo build system |
| Hot Reload | Yes — live reloads extension in Chrome on file save |
| Init | `npx plasmo init my-extension` |
| Build | `plasmo build` — outputs a production ZIP ready for Web Store |

### 1.2 Mapping Engine — MapLibre GL JS

MapLibre is a fully open-source fork of Mapbox GL JS with **zero licensing cost**. It renders hardware-accelerated vector tiles and can draw custom GeoJSON layers (e.g., a 300 m radius circle) inside a small popup iframe.

- Renders 3D building extrusions at any zoom level
- Accepts any tile source: OpenStreetMap, Maptiler free tier, Stadia Maps free tier
- Tiny footprint — bundles at ~250 KB gzipped
- Draw a radius circle:
  ```js
  turf.circle([lng, lat], 0.3, { units: 'kilometers' })
  // → add as a GeoJSON fill layer in MapLibre
  ```

### 1.3 UI & Styling — TailwindCSS in Shadow DOM

Plasmo's content-script injection automatically wraps your popup in a **Shadow DOM boundary**. Your Tailwind utility classes (e.g., `bg-gray-900`, `text-emerald-400`) are completely isolated from the host page's stylesheet — NoBroker, MagicBricks, or any portal's CSS cannot leak in or out.

- Use `@plasmo/content-script` with `shadowMode: 'open'` in your config
- Tailwind is compiled to a scoped CSS string and injected into the shadow root — Plasmo handles this automatically
- Use `dark:` variants freely — shadow DOM means no conflict with host-page dark-mode overrides

## Block 2 — Hyperlocal & Geospatial APIs

All data is fetched on-the-fly using a property's coordinates as the query anchor. No bulk scraping. No static datasets. Everything is real-time and free at MVP scale.

### 2.1 API Reference Table

| Data Metric | Source / API | Free Limit | Returns |
|---|---|---|---|
| Local Markets & Shops | Overpass API (OpenStreetMap) | Unlimited (fair-use) | Nodes within radius — name, hours, type |
| Traffic Congestion | TomTom Traffic Flow API | 2,500 req/day | currentSpeed, freeFlowSpeed, confidence |
| Points of Interest | Overpass API + Wikidata | Unlimited | Landmarks, transit, parks — sorted by distance |
| Address Geocoding | Photon API (Komoot) | Unlimited (self-hostable) | `[Lng, Lat]` decimal from text address |

### 2.2 Overpass API — Query Pattern

```
[out:json][timeout:10];
(
  node[shop=supermarket](around:300,{LAT},{LNG});
  node[amenity=pharmacy](around:300,{LAT},{LNG});
  node[amenity=marketplace](around:300,{LAT},{LNG});
);
out body;
```

> **Endpoint:** `POST https://overpass-api.de/api/interpreter` — no auth required.

### 2.3 TomTom Traffic Flow API — Request Pattern

```
GET https://api.tomtom.com/traffic/services/4/flowSegmentData/absolute/10/json
  ?point={LAT},{LNG}&key={TOMTOM_API_KEY}
```

> Returns: `currentSpeed` (km/h), `freeFlowSpeed` (km/h), `currentTravelTime` (seconds). Delta = congestion score.

### 2.4 Photon Geocoding — Request Pattern

```
GET https://photon.komoot.io/api/?q=Koramangala+5th+Block+Bengaluru&limit=1&lang=en
```

> Returns: GeoJSON FeatureCollection. Extract `features[0].geometry.coordinates` → `[lng, lat]`.

## Block 3 — Backend & Database

The backend is intentionally minimal. Only structured, user-contributed data that cannot be derived from public APIs lives here. The entire schema fits comfortably inside Supabase's free tier.

### 3.1 Platform Choice — Supabase vs Appwrite

| Criterion | Supabase | Appwrite |
|---|---|---|
| Free DB Storage | 500 MB Postgres | 2 GB (generous) |
| Geospatial | ✅ Native PostGIS (`ST_DWithin`, `geography`) | ❌ No native PostGIS |
| Anonymous Auth | Built-in — one line of SDK code | Built-in — same ease |
| Row Level Security | Postgres RLS — very fine-grained | Role-based — simpler |
| **Recommendation** | ✅ **PREFERRED** (PostGIS support) | Good fallback if Supabase limits hit |

### 3.2 Database Schema

#### Table A — `water_reports`

| Column | Type | Purpose |
|---|---|---|
| `id` | `uuid` | Primary key, auto-generated |
| `building_slug` | `text` | Normalized building identifier (lowercase, no spaces) |
| `water_source` | `text` | `'Corporation'` \| `'Borewell'` \| `'Tanker'` \| `'Mixed'` |
| `quality_score` | `smallint` | 1–5 integer (1=poor, 5=excellent) |
| `anonymous_uid` | `uuid` | Supabase anon auth user ID — used for dedup |
| `submitted_at` | `timestamptz` | Auto-timestamp, used for 24h rate-limiting check |
| `coordinates` | `geography` | PostGIS point — enables `ST_DWithin` proximity queries |

#### Table B — `landlord_scorecards`

| Column | Type | Purpose |
|---|---|---|
| `id` | `uuid` | Primary key |
| `building_slug` | `text` | FK-style link to building identity |
| `safety_score` | `smallint` | 1–5 integer |
| `deposit_returned` | `boolean` | True/False self-reported |
| `maintenance_score` | `smallint` | 1–5 integer |
| `noise_score` | `smallint` | 1–5 integer |
| `anonymous_uid` | `uuid` | Submitter's anon ID |
| `submitted_at` | `timestamptz` | Timestamp for rate-limit enforcement |

### 3.3 Row-Level Security (RLS) Policies — Run These Before Anything Else

> ⚠️ **Critical:** Without RLS, your Supabase database is publicly writable by anyone with your anon key (which is visible in your extension bundle).

```sql
-- Allow anyone to READ
CREATE POLICY read_all ON water_reports
  FOR SELECT USING (true);

-- Allow INSERT only if user is authenticated (anon counts)
CREATE POLICY insert_authed ON water_reports
  FOR INSERT WITH CHECK (auth.uid() IS NOT NULL);

-- No UPDATE / DELETE policies = no client access for those operations
```

> Apply identical policies to `landlord_scorecards`.

## Block 5 — Architecture & Data Flow

### 5.1 Linear Data Flow

```
[WEB PAGE — Content Script]
  Plasmo content script reads the property address from the host page DOM
  using document.querySelector() on known listing selectors (MagicBricks, NoBroker, etc.)
       ↓
[GEOCODING LAYER]
  Text address → Photon API → [Lng, Lat] decimal coordinates.
  Result cached in chrome.storage.session to avoid re-geocoding on tab refresh.
       ↓
[PARALLEL FETCH LAYER — Promise.allSettled()]
  Three concurrent requests fire simultaneously:
    1. Overpass API  — shops & amenities within 300 m radius
    2. TomTom API    — current traffic speed delta on nearest road
    3. Supabase DB   — water_reports + landlord_scorecards for this building_slug
       ↓
[DATA AGGREGATION LAYER]
  Results merged into a single LiveabilityPayload object.
  Missing or errored APIs degrade gracefully (show '--' in UI, not a crash).
       ↓
[MAPLIBRE RENDER]
  Mini-map renders inside extension popup:
    (a) Property pin
    (b) 300 m GeoJSON circle
    (c) Shop icons from Overpass nodes
       ↓
[SCORECARD UI]
  Tailwind dark-theme dashboard displays the aggregated Liveability Index
  with individual category scores.
```

### 5.2 Anonymous Authentication Flow

```ts
// background.ts — runs once at extension install
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(SUPABASE_URL, SUPABASE_ANON_KEY);

chrome.runtime.onInstalled.addListener(async () => {
  const existing = await chrome.storage.local.get('sb_session');
  if (!existing.sb_session) {
    const { data } = await supabase.auth.signInAnonymously();
    await chrome.storage.local.set({ sb_session: data.session });
  }
});
```

The session token is silently generated on install, stored in `chrome.storage.local`, and reused on every subsequent API call. The user sees zero login screens.

### 5.3 Submission Rate-Limiting (Server-Side Check)

```sql
-- Supabase RPC — called before accepting any new submission
SELECT id FROM water_reports
WHERE building_slug = $1
  AND anonymous_uid  = auth.uid()
  AND submitted_at  > NOW() - INTERVAL '24 hours';

-- If rows > 0: reject with 429 Too Many Requests
```

> Wrap this in a Supabase Postgres Function (RPC) to keep client-side code clean.

### 5.4 Authentication Strategies Compared

#### Strategy 1 — Anonymous Authentication (Recommended)

Both Supabase and Appwrite support silent anonymous sign-in.

- **How it works:** On install, the backend generates a random crypto-token saved in `chrome.storage.local`
- **User experience:** Zero login screens, user types nothing, unaware it's happening
- **Developer benefit:** Every user has a unique ID — DB can enforce "one submission per building per 24 hours" per anonymous UID

#### Strategy 2 — Client-Side Hardware Locking

```ts
// On submission, save a timestamp locally
await chrome.storage.local.set({
  [`last_submit_${buildingSlug}`]: Date.now()
});

// On popup open, check and disable the submit button if < 86400000ms ago
const stored = await chrome.storage.local.get(`last_submit_${buildingSlug}`);
const blocked = stored && (Date.now() - stored < 86_400_000);
```

> ⚠️ **Limitation:** Bypassable by clearing extension storage in DevTools. Use as UX polish only — never as your real spam defense.

#### Strategy 3 — Cloudflare Turnstile (Invisible CAPTCHA)

- Database is open for reads/writes, but the submission form has an invisible Turnstile widget
- User clicks "Submit" → a spinner verifies they aren't a headless script → data goes through
- **Free:** 1 million challenges per month at $0

## 🔐 Security Audit

Every flaw is rated by severity and paired with a **zero-cost mitigation**.

---

### 🔴 CRITICAL — Fix Before Publishing

#### Flaw 1 — API Keys Exposed in Extension Bundle

**Risk:** Plasmo compiles your code into a `.zip` anyone can unpack (`chrome://extensions → Pack extension`). Any hardcoded `TOMTOM_API_KEY` is immediately readable by any developer who installs your extension.

**Fix:**
1. Use Plasmo's `.env.chrome` system for all keys
2. In the **TomTom Developer Portal**: lock your API key to HTTP Referer = `chrome-extension://YOUR_EXTENSION_ID`. Requests from any other origin are rejected
3. Your Supabase anon key being visible is acceptable **only if RLS is enabled** — the anon key is a public key by design; RLS is the actual security gate

---

#### Flaw 2 — Supabase Database Open to Public Writes

**Risk:** A fresh Supabase project has RLS disabled by default. Anyone with your anon key (visible in the bundle) can run arbitrary `INSERT`, `UPDATE`, `DELETE` commands against your entire database via the Supabase REST API with a simple `curl` command.

**Fix:** Run the SQL in [Block 3.3](#33-row-level-security-rls-policies--run-these-before-anything-else) immediately after creating your tables. Verify by making a DELETE request with only your anon key in Postman — it must be rejected.

---

### 🟠 HIGH — Fix Before Launch

#### Flaw 3 — Anonymous Token Theft / Cloning

**Risk:** The session token in `chrome.storage.local` is accessible via DevTools (`Application → Extension Storage`). A stolen token lets a bad actor submit spam under someone else's anonymous ID indefinitely.

**Fix:** The server-side 24h rate-limit RPC (Block 5.3) makes token theft almost useless — a cloned token can only submit once per day per building, same as a real user. Add Cloudflare Turnstile on the submission form as a second layer.

---

#### Flaw 4 — DOM Injection / XSS via Address Scraping

**Risk:** Your content script reads raw address strings from the host page DOM. A maliciously crafted listing page could inject a script payload into a DOM node. If your extension evaluates that string without sanitization (e.g., `innerHTML`, `eval`), it executes in the extension's content-script context.

**Fix:**
```ts
// ❌ NEVER do this
element.innerHTML = scrapedAddress;
eval(scrapedAddress);

// ✅ Always do this
const address = element.textContent; // textContent only, never innerHTML
const encoded = encodeURIComponent(address); // URL-encode before passing to geocoding API
```

---

### 🟡 MEDIUM — Fix Within First Week

#### Flaw 5 — chrome.storage.local Rate-Limit Bypass

**Risk:** Any user can clear extension storage in DevTools and bypass the client-side submission lock trivially. This is a known, intentional limitation.

**Fix:** Do not rely on `chrome.storage.local` as your primary spam defense. It is only a UX convenience layer. All real rate-limiting must be enforced server-side via the Supabase RPC (Block 5.3).

---

#### Flaw 6 — TomTom API Key Quota Exhaustion (DoS)

**Risk:** Since your TomTom key is visible in the bundle, a competitor can extract it and exhaust your 2,500 free daily requests by 12:01 AM, leaving your real users with 429 errors all day.

**Fix:**
1. Lock the key to your extension's Referer in TomTom dashboard (most important)
2. Cache TomTom responses in `chrome.storage.session` keyed by `[lat, lng]` rounded to 3 decimal places, with a 10-minute TTL

---

#### Flaw 7 — Overpass API IP Block

**Risk:** Firing an Overpass query on every scroll or hover event can generate hundreds of requests per minute from one user's IP. The Overpass team will temporarily block that IP.

**Fix:**
```ts
// Debounce all Overpass calls
const debouncedFetch = debounce(fetchOverpassData, 1000); // 1 second minimum gap

// Cache in session storage
const cacheKey = `overpass_${roundCoords(lat, lng)}`;
const cached = await chrome.storage.session.get(cacheKey);
if (cached[cacheKey]) return cached[cacheKey]; // return cache, skip API call
```

---

### 🟢 LOW — Address Before V2

#### Flaw 8 — Supabase Project Auto-Pause (Data Loss Risk)

**Risk:** Supabase free tier **pauses your project after 7 days of zero database activity**. During early MVP phase with no users, the project sleeps and incoming writes silently fail.

**Fix:** Set up a free cron job on [cron-job.org](https://cron-job.org) to ping your Supabase health endpoint every 5 days:
```
GET https://[project-ref].supabase.co/rest/v1/
Headers: apikey: YOUR_ANON_KEY
```

---

#### Flaw 9 — Cloudflare Turnstile Bypass (Sophisticated Bots)

**Risk:** Turnstile's invisible mode is effective against unsophisticated bots but can be bypassed by a determined attacker using real browsers controlled by Playwright/Puppeteer with stealth plugins.

**Fix:** Acceptable risk at MVP scale. For V2, add a Postgres trigger that auto-quarantines a `building_slug` from public averages if it receives more than 5 submissions within 1 hour:

```sql
CREATE OR REPLACE FUNCTION flag_suspicious_submissions()
RETURNS TRIGGER AS $$
BEGIN
  IF (SELECT COUNT(*) FROM water_reports
      WHERE building_slug = NEW.building_slug
        AND submitted_at > NOW() - INTERVAL '1 hour') > 5 THEN
    NEW.is_quarantined = TRUE;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### Developer Discipline Rules to Stay Free

1. **Cache aggressively.** Every external API response must be cached in `chrome.storage.session` with a TTL. Overpass: 5 min. TomTom: 10 min. Supabase reads: 2 min. This multiplies your effective free quota by 10–50x.

2. **Round coordinates.** Before using `[lat, lng]` as a cache key, round to 3 decimal places (~111 m precision). Two users 50 m apart reuse the same cache entry.

3. **Lazy-load APIs.** Don't fire all three APIs on page load. Geocode first. Only fire Overpass and TomTom when the user opens the extension popup. This cuts ~60% of API calls.

4. **Use `Promise.allSettled`, not `Promise.all`.** If TomTom fails, your UI should still render Overpass results. Never let one API failure crash the entire scorecard.

5. **Monitor your quotas weekly.** Bookmark TomTom Developer Portal → Usage Dashboard. Personal rule: if daily requests exceed 1,500 (60% of quota), add more caching before proceeding.

6. **Never store personal data.** No names, phone numbers, or review text in your DB. Integers and booleans only. This keeps you outside GDPR/DPDP Act compliance complexity entirely at MVP stage.
